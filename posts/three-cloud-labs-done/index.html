Three cloud labs, three architectures - what I built this semester

 Posted on May 24, 2026

This week I wrapped up the third laboratory work for the VGTU “Debesų kompiuterija” (Cloud Computing) course. Looking back, the three labs together form a neat tour of how modern web/game infrastructure is actually deployed - from screws-and-bolts manual provisioning, through containers, all the way to serverless static hosting. Here is a quick recap.

Lab 2 - “MMO Serveris”

The brief: stand up a private Dark Age of Camelot game server two ways and compare them.

Part A - manual install (Windows Server VM):

  • Created a Windows Server 2022 VM on Azure (Standard_D2s_v3, North Europe)
  • Installed MariaDB 10.11, HeidiSQL, .NET 10 runtime, DOLServer 2.2.9.3910
  • Patched C:\DOLServer\config\serverconfig.xml to use MySQL instead of SQLite, hard-coded RegionIP to the public IP, fixed an older game.dll version mismatch issue, dual-boxed two clients on the same Windows host
  • End to end: about 75 minutes of manual setup

Part B - Docker Compose (Ubuntu VM):

  • One cloud-init.yml that installs Docker, drops in a compose file, and brings up mariadb:10.6 + ghcr.io/opendaoc/opendaoc-core:latest
  • End to end: about 12 minutes, mostly just waiting for the gameserver container to compile its scripts on first start
services:
  gameserver:
    image: ghcr.io/opendaoc/opendaoc-core:latest
    container_name: opendaoc-server
    ports:
      - "10300:10300"
      - "10400:10400"
    environment:
      SERVER_NAME: "Armandas"
      DB_CONNECTION_STRING: "server=db;port=3306;database=opendaoc;userid=root;password=***;treattinyasboolean=true"

The Docker version was roughly 6x faster to bring up than the manual Windows install - and it is fully reproducible. The trade-off: OpenDAoC expected a v1.127 client while my lab client was v1109, which produced a receive-buffer overflow at login. Same kind of compatibility tax shows up in every “lift-and-shift to container” project I have seen.

Lab 3 - Hugo + GitHub Pages (this site)

Today’s lab was a much smaller, much cheaper architecture:

  • A Standard_B1s Ubuntu VM was supposed to host nginx with a custom HTML page, but B1s was capacity-restricted in every European region for my student subscription - I fell back to Standard_D2s_v3 in East US 2
  • Installed nginx + neofetch, replaced the default site with a custom index.html (“Simple static page by Armandas Kaleinykas…”), opened port 80 in the Azure NSG
  • Then jumped over to GitHub Pages: created two repos
    • Dason222/hugo - private, source files
    • Dason222/dason222.github.io - public, the rendered HTML, hooked into the source repo as a git submodule
  • Wired up the Beautiful Hugo theme, wrote this post and an About page, pushed
  • Tore down the Azure VM the same day - the GitHub Pages site is free to host forever

What surprised me

  1. B-series VMs are gold dust right now. Eight straight regions returned SkuNotAvailable - Capacity Restrictions for B1s/B1ms/B2s on my subscription. The lab PDF assumes B1s availability; in practice I had to move to the slightly more expensive D-series.

  2. az vm run-command invoke is a hidden superpower. No SSH, no key wrangling, just shell scripts shipped straight into the VM. The entire nginx + neofetch installation was one CLI call.

  3. Git submodules feel right for the Hugo + Pages workflow. The source repo can stay private while the rendered public/ folder is its own commit history pushed to the .github.io repo - a clean separation of “what I wrote” vs “what the world sees”.

What is next

The course continues, but my plan for the rest of the Azure for Students credit:

  • Migrate this blog to Azure Static Web Apps (still free, plus serverless Functions for forms/comments)
  • Put my OpenDAoC server on a B2ats_v2 so a couple of friends can actually connect

If you are also at VGTU and stuck on one of these labs, ping me on github.com/Dason222.
